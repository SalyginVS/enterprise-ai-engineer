# Конспект: Основы AI-Агентов на Python (M0)

В этом файле собраны пошаговые разборы всех скриптов из папки `src`, которые демонстрируют эволюцию от простого HTTP-запроса до полноценного ИИ-агента службы поддержки.

---

## 01: Базовый HTTP запрос (`01_raw_http.py`)
**Суть:** самый "сырой" запрос к LLM. Использование `requests` для отправки POST-запроса на сервер Ollama.
* Мы общаемся с моделью напрямую, отправляя JSON с историей сообщений.
* Ответ также приходит в формате JSON, откуда мы забираем `message -> content`.
* Это демонстрирует, что в основе любой ИИ-библиотеки лежит стандартный сетевой обмен данными.



## 02: Использование библиотеки (`02_ollama_library.py`)
**Суть:** тот же вызов, но через официальную библиотеку `ollama` для Python.
* Создается объект `client = Client(...)`.
* Запрос отправляется одной строкой через метод `.chat(model=..., messages=...)`.
* Вместо парсинга сырых словарей мы используем удобную "точечную" нотацию: `response.message.content`.

## 03: Статистика токенов (`03_token_stats.py`)
**Суть:** получение детальной статистики (метаданных) для контроля расходов (FinOps) и замеров скорости (бенчмаркинг).
* Из `response` извлекаются `eval_count` (сгенерированные токены) и `eval_duration` (потраченное время в наносекундах).
* Наносекунды делятся на `1_000_000_000`, чтобы получить секунды.
* Считается скорость в `tok/s` (токенов в секунду).

## 04: System Prompt и JSON Output (`04_system_prompt.py`)
**Суть:** задаем модели роль и требуем ответ в строгом машинном формате (JSON).
* **System Prompt:** отправляется отдельным сообщением с ролью `"system"`. Это "перепрошивка" модели.
* **temperature = 0.1**: параметр креативности снижается почти до нуля, чтобы ответы были стабильными и предсказуемыми.
* Полученный текст парсится (очищается от markdown-разметки ````json ````) и превращается в словарь через `json.loads()`.

## 05: Few-Shot Prompting (`05_few_shot.py`)
**Суть:** управление логикой модели через наглядные примеры вместо длинных словесных инструкций.
* В системном промпте прописываются пары `Input -> Output`, обернутые в теги `<example>`.
* Модель сама улавливает паттерн и начинает применять его к новым пользовательским запросам.
* Один хороший пример часто заменяет страницу сложного текста.

## 06: Цепочка рассуждений / Chain-of-Thought (`06_chain_of_thought.py`)
**Суть:** заставляем модель "подумать вслух" перед финальным ответом (CoT), чтобы повысить точность на сложных задачах.
* В промпте требуем отвечать строго внутри тегов `<thinking>` и `<answer>`.
* Скрипт разрезает ответ с помощью строковых методов (`split`) на две части: рассуждения и сам JSON.
* Пошаговое мышление сильно снижает риск ошибки (галлюцинаций) при классификации неоднозначных входных данных.

## 07: Smart Pipeline (`07_smart_pipeline.py`)
**Суть:** умный конвейер, комбинирующий дешевый и дорогой методы (Fast + CoT).
* **Эскалация**: сначала пробуем дешевый быстрый способ классификации. Если нейросеть вернула слишком низкий `confidence` (уверенность) или нарушила формат, скрипт автоматически отправляет запрос на дорогой метод CoT (Chain-of-Thought).
* Результаты упаковываются в удобный `@dataclass`, чтобы основному приложению было легко с ними работать (вместо обычных славарей `dict`).

## 08: Использование инструментов / Tool Calling (`08_first_tool.py`)
**Суть:** наделяем "мозг" языковой модели "руками", позволяя ей вызывать внешние функции (базы данных, калькуляторы и т.д.).
1. Мы передаем в запросе массив `tools` (описание функций в формате JSON Schema).
2. Модель вместо текста возвращает специальный ответ с ключом `tool_calls`.
3. Наш Python-скрипт ловит этот запрос, запускает реальную функцию (`execute_tool`), и отправляет результат как новое сообщение с ролью `"tool"`.
4. Модель читает результат и формирует финальный текстовый ответ пользователю.

## 09: Agent Loop (`09_agent_loop.py`)
**Суть:** превращаем скрипт в автономного Агента, который может решать сложные многошаговые задачи.
* Запускается цикл `for` (ограниченный `MAX_ITERATIONS` во избежание бесконечных зацикливаний).
* Агент думает $\rightarrow$ вызывает инструмент $\rightarrow$ анализирует ответ $\rightarrow$ решает: финальный ли это ответ, или нужно вызвать еще один инструмент (например, сначала узнать сумму заказа, а потом конвертировать её второй функцией).

## 10: Валидация аргументов (`10_validated_agent.py`)
**Суть:** защита приложения от галлюцинаций модели и пустых ответов (Robustness).
* **Pydantic:** мы создаем строгие схемы для входных данных инструментов с проверками `@field_validator`. Если модель придумает несуществующий номер счёта или "кривой" формат, валидатор выдаст ошибку.
* **Самокоррекция:** ошибка валидатора отправляется *обратно модели*, заставляя её исправиться и попробовать снова.
* **Fallback пустых ответов:** если цикл завершается, а модель забыла сказать текст человеку (пустой ответ), скрипт принудительно просит её дать ответ, отправляя скрытое сообщение.

## 11: Готовый Агент Поддержки / CustomerSupportAgent (`11_customer_support_agent.py`)
**Суть:** финальный аккорд — инкапсуляция всей логики в красивый ООП-класс.
* Агент умеет: искать в FAQ, проверять статусы заказов, классифицировать запросы.
* Наружу "торчит" только один простой метод `.handle(message)`, всё остальное скрыто (приватные методы с префиксом `_`).
* Обертка включает встроенную телеметрию (подсчет запросов, ошибок, потраченных итераций).
